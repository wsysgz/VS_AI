# VS_AI Phase 2 Executive Briefing Presentation Design

Date: 2026-04-10
Topic: `VS_AI` second-round presentation upgrade for WeChat, Telegram, and Markdown executive briefs
Status: Approved in conversation, written for planning

## 1. Goal

Upgrade the current text outputs so they read like a disciplined morning intelligence brief instead of a raw pipeline dump.

This phase should:

- keep the existing Beijing `07:00` operating model
- keep PushPlus / ClawBot as the short WeChat-facing channel
- keep Telegram as the full-text delivery channel
- keep Markdown as the repository-canonical archive
- improve readability, hierarchy, and consistency across all three text surfaces
- avoid coupling presentation work to a risky rewrite of the staged AI pipeline

The desired reading feel is:

- WeChat: `30-second morning judgment`
- Telegram: `forwardable executive brief`
- Markdown: `formal archive that remains easy to review later`

## 2. Approved Scope Decisions

The following decisions are approved for this phase:

1. This phase focuses on `WeChat / Telegram / Markdown` only.
2. A standalone HTML briefing page is explicitly out of scope for now and can be added later.
3. The presentation style should follow an `executive brief` direction, while absorbing some of the compact rhythm of the current morning-card style.
4. The staged AI pipeline remains the content engine; this phase should add a presentation adaptation layer rather than redesign the AI stages.
5. PushPlus and Telegram should stop sharing one compromise template and instead render from the same brief structure at different lengths.
6. The archive Markdown remains the canonical long-form artifact in the repository.

## 3. Problem Statement

The current first-round output is already richer than the original log-style report, but it still has three presentation weaknesses:

1. WeChat, Telegram, and Markdown do not yet feel like three layers of one system.
2. Message text still carries some `pipeline output` flavor instead of a stable `briefing product` voice.
3. Different channels are too close to one shared rendering path, which makes future polishing harder.

The project now needs a clear presentation architecture before moving into wider source expansion and stronger AI reasoning.

## 4. Non-Goals

This phase does not:

- redesign the analysis / summary / forecast AI stages
- introduce HTML, screenshots, image cards, or browser-rendered delivery
- expand information sources
- change scheduling, delivery channels, or authentication behavior
- replace the existing report JSON archive shape

Those remain later work.

## 5. Alternatives Considered

### Option A: Minimal copy rewrite

Only change headings, labels, and section order in the current renderers.

Pros:

- fastest path
- lowest risk

Cons:

- does not establish a reusable presentation layer
- HTML briefing would still need a separate restructuring later
- channel divergence would remain awkward

### Option B: Unified brief skeleton on top of current payload

Keep the current `summary_payload`, but add a dedicated adaptation layer that converts it into stable briefing blocks before rendering.

Pros:

- creates one consistent content model for all text channels
- avoids destabilizing the AI pipeline
- gives later HTML work a clean input contract

Cons:

- slightly more code than a straight copy rewrite

### Option C: Fully independent channel-specific layouts

Let WeChat, Telegram, and Markdown each build their own structure directly from raw payload fields.

Pros:

- each channel can be optimized locally

Cons:

- highest maintenance burden
- high risk of drift in tone and structure
- poor foundation for future HTML or richer views

## 6. Chosen Design

This phase adopts **Option B**.

The system should add a lightweight `brief composer` between `summary_payload` and the output renderers.

Target flow:

`analysis + ai outputs -> summary_payload -> briefing composer -> brief blocks -> channel renderers -> delivery / archive`

This keeps content generation and presentation clearly separated.

## 7. Briefing Architecture

### 7.1 Source Of Truth

`summary_payload` remains the upstream content contract generated by the current pipeline.

Important existing fields to preserve and reuse:

- `one_line_core`
- `executive_summary`
- `key_points`
- `key_insights`
- `analyses`
- `forecast`
- `limitations`
- `actions`
- `meta`
- `stage_status`

### 7.2 New Adaptation Layer

Add a new module that converts the current payload into a stable executive-brief shape.

Recommended file:

- `src/auto_report/pipeline/briefing.py`

Its job is not to invent new facts. Its job is to normalize existing content into a stable, render-friendly structure.

### 7.3 Brief Blocks

The brief composer should produce one normalized structure with fields similar to:

- `title`
- `generated_at`
- `judgment`
- `mainlines`
- `topic_briefs`
- `watchlist`
- `risk_note`
- `action_note`
- `limitations`
- `stage_status`
- `meta`

Recommended meanings:

- `judgment`: the top one-line conclusion, usually from `one_line_core`
- `mainlines`: the top 2-3 major themes, usually adapted from `key_points`
- `topic_briefs`: compact selected topic summaries adapted from `analyses`
- `watchlist`: the short-term observation line, usually adapted from `forecast.most_likely_case`
- `risk_note`: the most important uncertainty or limitation
- `action_note`: the most useful next action or follow-up line

### 7.4 Fallback Rules

The brief composer must be conservative:

- if `key_points` is sparse, it should derive fewer `mainlines` instead of inventing filler
- if `analyses` is sparse, it should produce fewer `topic_briefs`
- if `forecast` is weak or missing, `watchlist` should fall back to a cautious observation line
- if `limitations` is empty, `risk_note` can be omitted instead of fabricated

This phase should improve reading quality, not hide uncertainty.

## 8. Channel-Specific Rendering Contracts

### 8.1 WeChat / PushPlus Short Brief

Purpose:

- deliver a `30-second morning judgment`
- remain easy to scan in a phone notification or chat window

Shape:

1. title and timestamp
2. `今日判断`
3. `三条主线`
4. `一个提醒`
5. `一个观察`
6. detail link

Style rules:

- keep paragraphs short
- one thought per line
- prefer strong labels and short rhythm
- avoid long topic dumps

### 8.2 Telegram Full Brief

Purpose:

- deliver a forwardable full-text briefing with better depth than WeChat

Shape:

1. title and timestamp
2. `今日判断`
3. `执行摘要`
4. `关键主线`
5. `重点主题`
6. `短期观察`
7. `局限与提醒`
8. detail link

Style rules:

- more complete than WeChat, but still not as dense as Markdown
- topic section should be compact and selective
- readable after automatic splitting into `<=4096` character chunks

### 8.3 Markdown Archive

Purpose:

- remain the repository-authoritative long-form version
- support later review, debugging, and handoff

Shape:

1. title and timestamp
2. `一句话判断`
3. `执行摘要`
4. `关键洞察`
5. `重点主线`
6. `重点主题分析`
7. `短期推演`
8. `局限性`
9. `行动建议`
10. optional run meta

Style rules:

- must read like a stable report, not raw diagnostics
- can include more analysis detail than Telegram
- should remain plain Markdown for robustness

## 9. File Responsibilities

### 9.1 New Module

- `src/auto_report/pipeline/briefing.py`

Responsibilities:

- normalize `summary_payload`
- build the shared brief blocks
- keep presentation preparation separate from AI generation

### 9.2 Existing Modules To Change

- `src/auto_report/outputs/renderers.py`
  - render WeChat short brief from brief blocks
  - render Telegram full brief from brief blocks
  - render Markdown archive from brief blocks plus supporting detail

- `src/auto_report/app.py`
  - route PushPlus and Telegram through separate renderer entry points
  - keep Markdown archive generation aligned with the same brief structure

Potentially minor changes only:

- `src/auto_report/pipeline/analysis.py`
  - only if needed to ensure the payload carries the minimum fields expected by the composer

## 10. Testing Strategy

### 10.1 New Tests

- `tests/test_briefing.py`

  - verifies sparse inputs degrade gracefully

### 10.2 Updated Tests

- `tests/test_renderers.py`

  - verifies Telegram output uses the full brief shape
  - verifies Markdown output contains the new briefing structure

  - verifies run-once still generates the expected files

Required local verification:

1. `python -m pytest -q`
2. `python -m auto_report.cli run-once`
3. inspect:
   - `data/reports/latest-summary.md`
   - `data/reports/latest-summary.json`
   - `data/state/run-status.json`
4. if local push remains enabled, verify that PushPlus and Telegram text shape did not regress

## 11. Risks And Tradeoffs

- If the composer over-normalizes, the report may start sounding generic; the implementation should preserve the project's current factual density.
- If Telegram and WeChat diverge too far, later maintenance becomes expensive; that is why both should be driven from the same brief blocks.
- If Markdown becomes too long, Telegram splitting will remain the only readable version; the Markdown renderer should stay structured and selective.
- Later HTML work should reuse the same brief blocks instead of bypassing them.

## 12. Completion Criteria

This phase is complete when:

- WeChat, Telegram, and Markdown clearly read like different layers of one executive briefing system
- PushPlus no longer reuses a generic fallback-style text body
- Telegram no longer depends on raw Markdown plus a pasted link
- Markdown archive reads like a formal brief rather than a pipeline dump
- all local verification steps pass without regressing the current run model

## 13. Recommendation Summary

The right second-round presentation upgrade is not to jump straight into HTML.

It is to first define a reusable executive-brief structure for text channels:

- one shared brief composer
- one shared content language
- three channel-specific depths
- no unnecessary AI pipeline rewrite

That creates a stable base for the next work:

- HTML briefing pages
- stronger source expansion
- stronger AI reasoning quality
