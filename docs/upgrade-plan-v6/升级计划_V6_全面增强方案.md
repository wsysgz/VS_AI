### 1.4 关键发现

1. **AI Analysis 串行调用是绝对最大瓶颈** — 占总耗时 55%+, 波动最大(取决于候选数量)
2. **GitHub Actions timeout=25min 不够用** — 正常情况就可能超时，必须拆分 job
3. **采集层全部串行** — RSS/GitHub/Website 三大类之间及内部都无并发
4. **无执行计时** — 不知道每个阶段到底花了多久，无法精准优化
5. **单模型依赖 DeepSeek** — API故障则全线降级到规则fallback

### 3.1 Workflow Job 拆分（核心改动）

**现状问题**: 所有步骤在一个 job 中顺序执行, timeout=25min 已经是极限

**文件**: `cli.py` / `app.py`

新增三个 CI 专用子命令，将现有 `run_once` 拆为三个可独立执行的阶段：

### 3.3 采集层并发化

**文件**: `sources/collector.py`
**技术**: `concurrent.futures.ThreadPoolExecutor`

改动清单：

1. **三大类采集器并行启动** (RSS / GitHub / Website 同时跑)
2. **GitHub 内部仓库并行** (9个仓库同时 API 调用)
3. **Website 内部并行** (12个网站同时抓取)
4. **全局超时控制** (整个采集不超过120秒)

核心改造：

```python
# collector.py 重构后
import concurrent.futures

def collect_all_items(settings):
    all_items = []
    diagnostics = []

    # 三大类采集器并行启动
    with concurrent.futures.ThreadPoolExecutor(max_workers=3) as executor:
        future_to_name = {
            executor.submit(_collect_rss, settings): "rss",
            executor.submit(_collect_github, settings): "github",
            executor.submit(_collect_websites, settings): "websites",
        }
        for future in concurrent.futures.as_completed(future_to_name, timeout=120):
            name = future_to_name[future]
            try:
                items, msgs = future.result()
                all_items.extend(items)
                diagnostics.extend(msgs)
            except Exception as exc:
                diagnostics.append(f"{name} collection failed: {exc}")

    return all_items, diagnostics


# _collect_github 内部也并发化:
def _collect_github(settings):
    # ... 初始化 ...
    def _fetch_one_repo(full_name):
        resp = requests.get(f"https://api.github.com/repos/{full_name}", timeout=20, headers=headers)
        resp.raise_for_status()
        return normalize_github_repository_detail(...)

    with concurrent.futures.ThreadPoolExecutor(max_workers=5) as executor:
        futures = {executor.submit(_fetch_one_repo, repo): repo for repo in repositories}
        for future in concurrent.futures.as_completed(futures, timeout=60):
            try:
                item = future.result()
                if item: items.append(item)
            except Exception as exc:
                diagnostics.append(f"GitHub repo failed: {exc}")

# _collect_websites 同理加 ThreadPoolExecutor(max_workers=6)
```

**性能预估**：

| 场景             | 旧(全串行) | 新(并发化)            | 加速比         |

| 采集总计         | 3-8 min    | **1-2.5 min**   | **3-4x** |
| GitHub API (9仓) | 2-5 min    | **0.5-1 min**   | ~5x            |

## 四、Phase 1: AI Pipeline 性能优化 🚀 最大收益

**目标**: 将 AI Analysis 阶段从 10-25min 降到 3-8min
**预计工作量**: 半天
**预期收益**: 总运行时间降低 40-60%

### 4.1 并行 Analysis 调用（核心优化）

**文件**: `pipeline/ai_pipeline.py`
**现状**: 第196行 `for candidate in selected_candidates:` 逐个同步调 DeepSeek
**改动**: `ThreadPoolExecutor` 并行，最多5路同时调用

```python
from concurrent.futures import ThreadPoolExecutor, as_completed

def _analyze_single_candidate(candidate, ai_readings):
    """单个候选的AI分析（线程安全，可被并发调用）"""
    prompt = _build_analysis_prompt(ai_readings["analysis"], candidate)
    raw = summarize_with_deepseek(prompt)
    parsed = _parse_json_block(raw)
    parsed = _unwrap_named_payload(parsed, "analysis")
    parsed = _validate_required_keys(parsed, ANALYSIS_REQUIRED_KEYS)
    parsed.setdefault("title", candidate.title)
    parsed.setdefault("url", candidate.url)
    parsed.setdefault("primary_domain", candidate.primary_domain)
    return parsed


# 替换 run_staged_ai_pipeline 中的原有 for 循环:
if ai_enabled:
    max_workers = min(len(selected_candidates), 5)  # 最多5路并发
    analyses = []

    with ThreadPoolExecutor(max_workers=max_workers) as executor:
        future_to_candidate = {
            executor.submit(_analyze_single_candidate, c, ai_readings): c
            for c in selected_candidates
        }
        for future in as_completed(future_to_candidate):
            candidate = future_to_candidate[future]
            try:
                analysis = future.result(timeout=60)  # 单个超时60s
                analyses.append(analysis)
            except Exception:
                analyses.append(_fallback_analysis(candidate))  # 独立fallback

    stage_status["analysis"] = "ok"
else:
    analyses = [_fallback_analysis(c) for c in selected_candidates]
    stage_status["analysis"] = "fallback"
```

**性能预估**：

| 候选数 | 旧(串行) | 新(5路并行)      | 节省   |
| ------ | -------- | ---------------- | ------ |
| 5 个   | 100-200s | **20-40s** | 75-80% |
| 8 个   | 160-320s | **32-64s** | 80%    |
| 12 个  | 240-480s | **48-96s** | 80%    |

**注意事项**：

- `max_workers=5` 平衡速率和 DeepSeek rate limit
- 每个线程独立 fallback，一个失败不阻塞其他
- 单个请求 timeout=60s（比原来 40s 宽松一些，因为有 retry）

### 4.2 AI 预筛选（参考 Horizon 的 Score+Filter）

**原理**: 在送去完整三阶段分析前，先用一次轻量 LLM 调用批量打分过滤低质量内容
**新增文件**: `pipeline/scoring_llm.py`

核心逻辑：

```
候选主题(12个) → LLM批量打分(1次调用, ~15s) → 分数 ≥ 5.0 的进入深度分析(约6个)
                                                    ↓ 分数 < 5.0 的走 fallback
```

**集成位置**: `run_staged_ai_pipeline()` 入口处，在 Analysis 循环之前

**收益**：

| 指标                   | 无预筛选                  | 有预筛选                         |
| ---------------------- | ------------------------- | -------------------------------- |
| LLM 总调用次数(12候选) | 14次 (12分析+1汇总+1预测) | ~9次 (1打分+~6分析+1汇总+1预测) |
| Token 消耗             | ~100K tokens              | ~55K tokens (**-45%**)     |
| Analysis 时间          | 160-320s                  | **35-55s** (含打分)        |
| 报告质量               | 含噪音                    | 更聚焦高价值信号                 |

### 4.3 DeepSeek 客户端加固

**文件**: `integrations/deepseek.py`

三项改进：

1. **Session 连接池复用** — TCP 保持连接，减少每次 ~100ms 握手开销
2. **Retry + 指数退避** — 网络抖动自动恢复 (3次重试: 1s→2s→4s)
3. **429 Rate Limit 处理** — 被限流时等待 60s 后重试
4. **递增超时** — 第1次 45s, 第2次 55s, 第3次 65s

| -------------- | --------------------------------------- | -------------- | ----------------- |

| AI 预筛选      | **-60~180s** (减少调用数)         | **-45%** | 中 (新增一个文件) |

### 3.3 高风险信号 Issue 告警

**场景**: forecast 阶段检测到高风险信号时, 自动创建 GitHub Issue 通知人工复核

**实现方式**: `actions/github-script@v7` + 条件触发

```yaml
  - name: Check risk level
    id: check-risk
    run: echo "level=$(python -c "import json; print(json.load(open('data/state/run-status.json')).get('risk_level','low'))")" >> $GITHUB_OUTPUT

  - name: Create issue on high risk
    if: steps.check-risk.outputs.risk_level == 'high'
    uses: actions/github-script@v7
    with:
      script: |
        github.rest.issues.create({
          owner: context.repo.owner,
          repo: context.repo.repo,
          title: '🔴 高风险信号检测: ' + new Date().toISOString().split('T')[0],
          body: '自动检测到高风险信号，请人工复核。\n\n[查看报告](https://wsysgz.github.io/VS_AI/)'
        })
```

### 4.1 执行计时仪表盘

**目标**: 每个阶段记录精确耗时, 写入 run-status.json

```python
class StageTimer:
    """简易阶段计时器"""
    def __init__(self, name: str):
        self.name = name
        self.start = 0
        self.elapsed = 0

    def __enter__(self):
        self.start = time.time()
        return self

    def __exit__(self, *args):
        self.elapsed = time.time() - self.start
```

在 `app.py` 各阶段使用:

```python
timings = {}

with StageTimer("collection") as t:
    items, diagnostics = collect_all_items(settings)
timings["collection"] = round(t.elapsed, 1)

with StageTimer("ai_analysis") as t:
    # ... AI pipeline ...
timings["ai_analysis"] = round(t.elapsed, 1)

# 最终写入 run-status.json
status["timings"] = timings
```

**产出示例** (`run-status.json`):

```json

  "timings": {
```

## 十、风险与应对

| 风险                              | 影响                    | 概率 | 应对措施                                          |
| --------------------------------- | ----------------------- | ---- | ------------------------------------------------- |
| DeepSeek 并发 rate limit          | 分析失败降级到 fallback | 中   | max_workers=5 + 429自动retry + 每线程独立异常处理 |
| GitHub Actions 免费额度耗尽       | 月度 2000min 可能不够   | 低   | Phase 0 优化后预计每天 ≤ 45min, 月耗 ~1350min    |
| Website 源反爬/结构变更           | 采集失败                | 中   | try/except 容错 + 诊断日志 + 定期维护源配置       |
| HN/新源引入噪音                   | 报告质量下降            | 低   | AI预筛选(≥5分阈值) + 人工review初期输出          |
| Workflow 拆分后 artifact 传递失败 | 后续 job 无法运行       | 低   | GitHub Actions artifact 是成熟机制, 失败率极低    |
