# V1升级方案

> 适用仓库：`D:\GitHub\auto`
>
> 最后更新：2026-04-18

这份方案不是操作手册，而是 VS_AI 的阶段性路线图。它回答的是：接下来几年别把工程做歪，当前 V1 到底先做什么、后做什么。

## 1. V1 的定义

对当前仓库来说，V1 不是“推倒重来重构一遍”，而是：

> **保留现有 `collect -> analyze -> render -> publish` 主链路，把来源稳定性、AI 可控性、运行可观测性补成一个可持续维护的系统。**

具体表现为：

- 来源入口不再主要依赖脆弱网页 listing
- AI 调用不再只是“能用”，而是可切换、可观测、可治理
- 运行态不再靠猜，而是统一回到 `run-status.json`
- 文档、交接、验收路径稳定，换人或失忆后能快速恢复

## 3. 为什么当前优先级是 P1

当前系统最痛的不是“没有更多功能”，而是：

1. 来源入口脆弱，listing 站点容易改版、超时、重定向、失效
2. 来源治理虽然已经有清单，但治理动作还没真正落地
3. 每次来源坏掉，都会放大对 AI、页面、推送、运营的连锁影响

这就是为什么当前应该先做 P1，而不是直接开始大规模服务化或编排层重构。

## 5. 分阶段路线图

### P0：基础收口（已完成）

目标：先把“系统能跑、能看、能验证、能交接”稳住。

已完成：

- PushPlus / ClawBot 最终送达验证
- 文档收口
- 验证基线固定
- 来源治理基础设施铺底

### P1：来源稳定性升级（阶段性收口）

目标：把“反复坏掉的网页来源”从被动修修补补，升级成可治理的来源体系。

工作包：

1. 建立来源分层
   - `stable-feed`
   - `fragile-listing`
   - `dynamic-site`
   - `manual-watch`
2. 给高价值脆弱来源优先寻找 RSSHub route 或稳定 feed
3. 给无稳定 feed 的来源建立 changedetection 旁路
4. 为 manual-review 来源补明确替换方案
5. 把治理候选转成真正可执行的运维清单

P1 的退出条件建议：

- 至少一批高价值 fragile source 已切到稳定 feed / RSSHub
- changedetection watch 清单已建立并有可追踪记录
- manual-review 来源都已有明确去向
- `source_governance` 能直接指导值班决策，不只是列数据

当前判断（2026-04-18）：

- [x] 高价值 fragile source 已完成第一轮稳定入口迁移
- [x] changedetection watch 清单已正式落表
- [x] manual-review 来源已经全部有明确去向或替代方向
- [x] `source_governance` 已强化为更直接的值班优先级视图
- [ ] `anthropic-news` 仍需远端 / 外网环境下完成 RSSHub 或官方 feed 最终验证

### P2：AI 层升级

目标：把当前 AI 调用从“能用”升级成“可控、可切换、可回溯”。

当前已知起点：

- `src/auto_report/integrations/llm_client.py` 已支持 OpenAI-compatible provider
- 当前已确认可直接切换的 provider：
  - DeepSeek 官方：`AI_PROVIDER=deepseek`
  - MiniMax-M2.7（通过 OpenAI-compatible 第三方入口）：`AI_PROVIDER=minimax_svips`
- 当前已完成能力：
  - `analysis / summary / forecast` 已支持按阶段切换不同 provider
  - 预筛选阶段默认复用 `analysis` 的 provider 设置
- 当前 GitHub reusable workflows 已支持读取统一的
  `AI_PROVIDER / AI_BASE_URL / AI_MODEL / AI_API_KEY` 配置
  - 默认远端行为仍保持 DeepSeek
  - 当 Repository Variables / Secrets 配置完成后，线上远端也可读取
    `ANALYSIS_* / SUMMARY_* / FORECAST_*` 的 stage-level provider 配置

推荐工作包：

1. 接入 LiteLLM Gateway
2. `llm_client` 支持直连 / Gateway 切换
3. 接入 Langfuse traces
4. 把 prompt eval 与线上 run 打通统一 trace 维度
5. 增加 provider fallback、budget guardrail、per-stage latency / token budget
6. [x] 支持 stage-level model routing（例如 `analysis=DeepSeek`, `summary=MiniMax`, `forecast=DeepSeek`）
7. [x] 让 GitHub Actions 读取统一的 `AI_PROVIDER / AI_BASE_URL / AI_MODEL / AI_API_KEY` 配置

P2 中关于双 AI 并存的建议默认分工：

- `analysis` / `pre_filter`
  - 优先使用更稳、成本更可控的 provider（当前建议 DeepSeek）
- `summary`
  - 优先使用更擅长综合整理和最终表达的 provider（可评估 MiniMax-M2.7）
- `forecast`
  - 初期可继续沿用 DeepSeek，后续再根据真实效果决定是否切给 MiniMax

这样做的目标不是“同时开两个模型就算升级”，而是：

- 让高频阶段优先考虑成本与稳定性
- 让低频但更看重表达质量的阶段使用更合适的模型
- 在不重写主链路的前提下，把双 AI 并存变成可配置、可观测、可回滚的能力

GitHub / Repository secrets & variables 规划建议：

- Repository Secrets
  - `DEEPSEEK_API_KEY`
  - `AI_API_KEY`（给 MiniMax 或未来第三方 OpenAI-compatible provider）
- Repository Variables
  - `AI_PROVIDER`
  - `AI_BASE_URL`
  - `AI_MODEL`
- Stage-level Repository Variables
  - `ANALYSIS_AI_PROVIDER`
  - `ANALYSIS_AI_BASE_URL`
  - `ANALYSIS_AI_MODEL`
  - `SUMMARY_AI_PROVIDER`
  - `SUMMARY_AI_BASE_URL`
  - `SUMMARY_AI_MODEL`
  - `FORECAST_AI_PROVIDER`
  - `FORECAST_AI_BASE_URL`
  - `FORECAST_AI_MODEL`
- Stage-level Repository Secrets
  - `ANALYSIS_AI_API_KEY`
  - `SUMMARY_AI_API_KEY`
  - `FORECAST_AI_API_KEY`
- 推荐分层：
  - 默认远端主线保持 `DeepSeek`
  - 当需要双 AI 分工时，再增量开启 stage-level routing

## 6. 推荐路线对比

### 方案 A：保守修补

只做：

- 继续手工修 selector
- 继续补个别来源
- 补一点 delivery retry

优点：

- 改动小

缺点：

- 来源脆弱性不会本质改善
- 后面仍会反复返工

### 方案 B：均衡偏激进（推荐）

核心思路：

- 保留现有 VS_AI 主链路
- 外挂来源侧车 + AI 网关 + tracing
- 优先把脆弱来源迁到更稳的入口

即：

```text
Current VS_AI core
+ RSSHub sidecar
+ changedetection sidecar
+ selective Firecrawl fallback
+ LiteLLM gateway
+ Langfuse tracing / evals
```

这是当前最值得执行的路线，因为它见效快、可回滚、和现有工程兼容。

### 方案 C：全面重构

上来就：

- 改成 Dagster orchestration
- 拆 collector / analyzer / publisher service
- 上数据库和任务队列

风险：

- 当前收益 / 成本比不高
- 很容易把注意力从“来源稳定性”和“可观测性”转移走

## 7. 外部项目整合建议

### 适合优先落地的

- RSSHub
  - 用作脆弱网站的稳定 feed 替代
- changedetection.io
  - 用作无 feed 站点的旁路监控
- LiteLLM
  - 用作统一 AI Gateway
- Langfuse
  - 用作 tracing / prompt 管理 / 成本与延迟观测

### 适合选择性使用的

- Firecrawl
  - 只给极难抓取的高价值站点做 fallback extractor
- OpenCLI
  - 更适合做来源侧车 / 运维侧车，尤其是无 feed、强 JS、需要浏览器登录态的复杂站点

### 适合借鉴方法但暂不直接接入的

- CLI-Anything
  - 适合作为 adapter registry / agent-native CLI 组织方式的参考
  - 当前不建议直接并入 VS_AI 主链，避免把项目重心从情报流水线转移到工具生态搭建

### 适合后置评估的

- Dagster
  - 适合 V1.5 / V2，而不是现在立刻迁移

## 8. 风险与回滚原则

### 风险

- 新侧车组件会增加部署复杂度
- 来源替换会带来短期覆盖波动
- LiteLLM / Langfuse 接入初期会增加配置量

### 回滚原则

- 所有新增组件优先 sidecar 化，不要直接替换主链路
- 先做配置级切换，不做一步式大重构
- 每个阶段都要能单独验证、单独回滚

## 9. 对 CLI-Anything / OpenCLI 的评估结论

### CLI-Anything

- [x] 已完成评估
- [x] 结论：当前阶段不建议直接引入主链
- [x] 价值：适合作为后续 adapter registry、skill packaging、agent-native tool catalog 的组织参考
- [ ] 当前不作为 P2 直接落地项

### OpenCLI

- [x] 已完成评估
- [x] 结论：比 CLI-Anything 更适合 VS_AI 当前阶段
- [x] 价值：适合作为来源侧车 / 运维侧车，用于浏览器登录态、复杂 JS 页面、可重复的 deterministic 抓取/验证
- [ ] 当前不进入主采集主链
- [ ] 建议作为 `P2.5 / P3` 的试点项，先选一个最难但高价值的动态来源做 pilot
