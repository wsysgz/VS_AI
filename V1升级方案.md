# V1升级方案

> 适用仓库：`D:\GitHub\auto`
> 
> 首版整理：2026-04-16

## 1. 目标

本方案的目标不是简单“继续修补日报脚本”，而是把当前 VS_AI 从“可用的工程化日报链路”升级为“可维护、可扩展、可观测、可持续清洗来源、可逐步自动优化”的 V1 系统。

本次方案强调三件事：

1. 因地制宜，优先复用现有工程，不推倒重来
2. 允许激进方案，但必须分阶段落地
3. 外部项目优先考虑能直接下载、自部署、低耦合集成的组件

## 2. 当前现状判断

### 2.1 已有基础

当前仓库已经具备：

- 多源采集、去重、评分、主题构建、AI 三阶段分析
- `auto` / `reviewed` 双轨发布
- PushPlus / Feishu / Telegram 三通道推送
- Pages 公开站、ops dashboard、review queue
- `run-status.json` 作为正式 observability 面
- 本地 workflow guard + pytest + prompt eval 验证链路

### 2.2 当前主要短板

1. 来源入口脆弱
   - 一部分来源靠手工配置网页列表页 + CSS selector，遇到改版、重定向、Cloudflare、410/404 就容易失效
2. 文档分散
   - 原先使用、运维、交接手册分散，维护成本高，知识同步慢
3. LLM 供应商接入较“点对点”
   - 当前 provider 接法够用，但缺少统一网关、预算、回退、速率治理能力
4. 生产观测仍偏轻量
   - 已有 `run-status.json`，但跨 run 的 tracing、prompt 版本管理、质量追踪仍不足
5. 来源清洗缺少系统机制
   - 目前更多是发现坏源后手工移除或禁用，缺少自动巡检/旁路替代方案

### 2.3 本轮已完成的即时修复

本轮已经先完成一批必要收口：

- PushPlus `clawbot` 最终送达验证接通

## 4. 对这些项目的整合判断

### 4.1 推荐直接落地的

#### A. RSSHub

推荐作为“来源补强侧车”直接接入。

适用场景：

- 官方博客有稳定站点，但现有列表页 selector 容易变
- 某些站点本身已有 RSS，只是当前抓取路径不稳定
- 希望减少手工维护 selector 的数量

接入方式：

- 新建 `config/sources/rsshub.yaml` 或在现有 `config/sources/*.yaml` 中增加 `mode: rss_feed`
- 为每个脆弱网站优先查 RSSHub route
- collector 优先走 RSS，失败再走当前网页采集或 fallback 抽取

#### B. changedetection.io

推荐作为“坏源旁路监控器”直接自托管。

它非常适合当前仓库里这类来源：

- 网站改版频繁
- 页面需要点击/接受 cookie
- 文章列表页经常返回空、超时、重定向

接入方式：

- 不替换主采集器，而是作为 sidecar
- 为高价值但高脆弱来源建立 watch
- watch 触发后写入一个轻量 feed / webhook / queue，由现有 pipeline 继续消费

#### C. LiteLLM + Langfuse 组合

这是当前仓库最值得做的“可观测 + 可切换 + 可控成本”升级。

整合方式：

- `llm_client` 不再直连某单一 provider，而是可选直连 LiteLLM Gateway
- LiteLLM 负责：
  - provider 统一
  - fallback
  - budget / rate limit
  - request logging
- Langfuse 负责：
  - traces
  - prompt version
  - eval
  - cost / latency dashboard

这套组合可以在不推翻现有 `ai_pipeline` 的前提下，显著提升生产能力。

### 4.2 推荐作为中期激进升级的

#### D. Firecrawl

推荐作为“复杂站点抽取兜底层”，但不要一上来全量替换当前采集器。

原因：

- 能力强，但复杂度和成本也更高
- 更适合处理：
  - JS 重页面
  - 深层链接探索
  - 需要 map / crawl 的新站

推荐姿势：

- 只给前 10 个高价值、最易失效的来源启用
- 作为 fallback extractor，不作为默认路径

#### E. Dagster

推荐作为 V1.5 / V2 编排升级候选，不建议现在立刻全面迁移。

原因：

- 当前仓库已有清晰 CLI 和 workflow 拓扑
- 现在最痛点不是“任务起不来”，而是“来源维护、观测和渠道稳态”
- 如果过早迁移编排层，会把注意力从真正收益更高的地方移开

## 5. 推荐路线：均衡偏激进方案

### 方案 A：保守修补

只做：

- 继续手工修 selector
- 更新文档
- 补一点 delivery 重试

优点：改动小。

缺点：来源脆弱性不会本质改善，后续还会反复返工。

### 方案 B：均衡偏激进（推荐）

核心思路：

- 保留现有 `collect -> analyze -> render -> publish` 主链路
- 外挂“来源侧车 + LLM 网关 + tracing 观测”三块新能力
- 逐步把脆弱来源迁到更稳的入口上

也就是：

```text
Current VS_AI core
+ RSSHub sidecar
+ changedetection sidecar
+ selective Firecrawl fallback
+ LiteLLM gateway
+ Langfuse tracing/evals
```

这是我最推荐的 V1 路线。

### 方案 C：全面重构

上来就：

- 改成 Dagster orchestration
- 拆 collector service / analyzer service / publisher service
- 上数据库和任务队列

优点：理论上更完整。

缺点：当前收益/成本比不高，风险太大，不适合作为 V1 起步。

## 6. V1 分阶段计划

### P0：本轮已做 / 立即收口

- PushPlus / ClawBot 最终送达验证
- 文档收口为统一入口
- 第一轮来源清洗
- 验证基线更新到 `200 passed`

### P1：来源稳定性升级（建议优先做）

目标：把“反复坏掉的网页来源”从被动修复变成系统治理。

工作包：

1. 建立来源分层
   - `stable-feed`
   - `fragile-listing`
   - `dynamic-site`
   - `manual-watch`
2. 为高价值脆弱来源优先寻找 RSSHub route
3. 给无稳定 RSS 的来源接入 changedetection 旁路
4. 给极难处理的来源接入 Firecrawl fallback
5. 为每个来源记录：
   - 最近成功时间
   - 失败类型
   - 连续失败次数
   - 建议替代方案

交付物：

- 一份来源分层清单
- 新的 source registry schema
- 一批替代入口配置
- `source_health` 更可行动的诊断信息

### P2：AI 层升级

目标：把当前 AI 调用从“能用”升级成“可控、可切换、可回溯”。

工作包：

1. 接入 LiteLLM Gateway
2. 将 `llm_client` 改成可切换：直连 / LiteLLM
3. 接入 Langfuse traces
4. 把 prompt eval 与线上 run 建立统一 trace 维度
5. 增加：
   - provider fallback
   - budget guardrail
   - per-stage latency and token budget

交付物：

- AI gateway config
- trace id / run id 对齐
- prompt version 管理流程
- 线上线下 eval 统一面板

### P3：运营与交付面升级

目标：让 VS_AI 从“有输出”变成“好管理、好追踪、好迭代”。

工作包：

1. 细化 ops dashboard
   - 来源故障排行榜
   - 渠道稳定性趋势
   - topic 覆盖趋势
   - prompt / model 成本趋势
2. review queue 增加优先级策略
3. Pages 增加：
   - 主题长期跟踪页
   - 来源健康页（内部可见）
   - reviewed / auto 差异提示
4. 引入 DailyOps 式日更优化任务
   - 每天自动巡检坏源
   - 每天自动刷新文档引用
   - 每天自动补测试和规则

### P4：V1.5 / V2 候选

- Dagster 编排
- collector / analyzer / publisher 拆服务
- 状态入库（SQLite/Postgres）
- 更强的多租户/多频道发布能力

## 7. 建议优先级排序

推荐你按这个顺序推进：

1. **来源稳定性升级**
2. **LiteLLM + Langfuse 接入**
3. **dashboard / review queue / Pages 升级**
4. **DailyOps 自动改良机制**
5. **Dagster / 服务化**

## 9. 风险与回滚策略

### 风险

- 新侧车组件增加部署复杂度
- 来源替换可能带来短期采集覆盖波动
- LiteLLM / Langfuse 接入初期会增加配置量

### 回滚原则

- 所有新增组件都应先以“旁路 / sidecar”方式接入
- 不替换现有主链路，直到新链路跑稳
- 配置级切换优先，避免一步式大重构

## 10. 我对 V1 的最终建议

如果只允许我给一个版本定义：

> **V1 = 保留现有 VS_AI 主链路，外接 RSSHub / changedetection / LiteLLM / Langfuse，优先解决来源脆弱性和 LLM 可观测性，逐步引入 DailyOps 式持续改良。**

这条路的好处是：

- 不浪费现有工程积累
- 可以快速见效
- 对后续更激进的 Dagster / 服务化升级兼容
- 每一步都能单独验证、单独回滚
