## 6.5 建库经验 / 防踩坑

- 先做 artifact，再做自动化；没有 artifact 的自动化几乎无法交接
- `candidate-updates.json` 是“接近执行的清单”，不是“无脑自动应用命令”
- 现在 `candidate-updates.json` 也会承载 `watch_update`；当前它应该尽量接近空队列，避免把已消化工作继续堆在里面
- `changedetection-watch-registry.json` 现在表示 repo 内 watch 状态；`planned/requested -> active_local -> blocked/unchanged/changed` 是当前推荐最小流转
- `watch-run-results.json` 是本地 watch 检查的最新结果，优先看 `blocked` 和 `changed`
- `apply_ready=false` 不是失败，它只是提醒我们当前还缺验证证据
- 这次 `anthropic-news` 的经验很典型：不要为了追求 RSS 形态去破坏一个已验证可用的 listing
- 远端 workflow 触发后不要一直前台 watch，记录 run 后继续本地推进；默认不需要再专门补看最终结果
- LiteLLM 本地接入时，先区分两类配置：
  - `litellm-config.local.example.yaml`：DeepSeek-only bootstrap
  - `litellm-config.example.yaml`：完整 stage alias 验证
- 如果发现 `summary/prefilter` 没走 alias，先查：
  1. `AI_API_KEY` / `LITELLM_MINIMAX_API_KEY` 是否可用
  2. `LITELLM_CONFIG_PATH` 是否误指向 local-only config
- Langfuse 首轮不要一上来全量上传 prompt / response；metadata-first 更稳，更适合先收工程可观测性
- `P3-B` 建库经验补充：
  - 飞书 API 在当前 profile 下可能不能：
    - 删除旧表
    - 删除旧 dashboard
    - 删除 dashboard block
    - 强制设置所有视图可见字段顺序
  - 更稳的做法是：
    - 能删就删
    - 删不掉就改名成“旧版…（停用）”
  - `status` 这类想做成单选字段时，直接用 `options` 顶层字段更稳
  - 仓库仍保持英文标准状态值，飞书可以显示中文状态，pull-back 时再映射回英文标准值
- 更安全的本地清理优先级是先删：
  - `.codex-pytest-temp*`
  - `.codex-tmp`
  - `.pytest_cache`
  - `.pytest-codex-*`
  - `.pytest-tmp`
  - `codex_pytest_temp`
  - `pytest-tempdir*`
- 不要把以下目录当成“临时垃圾”直接删：
  - `data/`
  - `docs/`
  - `out/`
  - `docs/superpowers/`
- 如果删除这类本地临时目录时遇到 `Access denied`，优先判断是不是测试进程残留或 ACL 异常；不要为了清理临时目录去误删正式交付目录
