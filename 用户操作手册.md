## 1. 3 分钟启动

```powershell
cd D:\GitHub\auto
python -m venv .venv
.venv\Scripts\Activate.ps1
pip install -r requirements.txt
pip install -e .
Copy-Item .env.example .env
```

推荐 provider 切换模板：

```powershell
# DeepSeek 官方（当前默认推荐）
$env:AI_PROVIDER='deepseek'
$env:AI_BASE_URL='https://api.deepseek.com'
$env:AI_MODEL='deepseek-chat'
$env:DEEPSEEK_API_KEY='<your-deepseek-key>'

# MiniMax-M2.7（通过 OpenAI-compatible 第三方入口）
$env:AI_PROVIDER='minimax_svips'
$env:AI_BASE_URL='https://api.svips.org/v1'
$env:AI_MODEL='MiniMax-M2.7'
$env:AI_API_KEY='<your-minimax-key>'
```

说明：

- `DeepSeek / MiniMax` 可以在同一台机器上并存。
- 当前已支持按阶段切换 provider。
- 预筛选阶段默认复用 `analysis` 的 provider 配置。

按阶段分工的推荐模板：


### 1.1 GitHub / Repository secrets 怎么配 MiniMax

如果你想让 GitHub Actions 未来也能切到 MiniMax，需要先在 GitHub 仓库里准备统一配置。

进入路径：

- GitHub 仓库
- `Settings`
- `Secrets and variables`
- `Actions`

建议准备：

**Repository secrets**

- `DEEPSEEK_API_KEY`
- `AI_API_KEY`  ← 给 MiniMax-M2.7 或其他第三方 OpenAI-compatible provider
- 保留已有推送 secrets：
  - `PUSHPLUS_TOKEN`
  - `PUSHPLUS_SECRETKEY`
  - `TELEGRAM_BOT_TOKEN`
  - `TELEGRAM_CHAT_ID`
  - `FEISHU_APP_ID`
  - `FEISHU_APP_SECRET`
  - `FEISHU_CHAT_ID`

**Repository variables**

- `AI_PROVIDER`
- `AI_BASE_URL`
- `AI_MODEL`

给 MiniMax-M2.7 的推荐值：

```text
AI_PROVIDER=minimax_svips
AI_BASE_URL=https://api.svips.org/v1
AI_MODEL=MiniMax-M2.7
```

注意：

- 当前 GitHub reusable workflows 还没有完全改成读取这组统一变量，仍然固定走 DeepSeek。
- 所以现在先把 secrets / variables 准备好即可；等 P2 代码改完，再把 workflow 切到统一 provider 配置。

## 2. 常用命令速查

### 2.1 生成当天日报

```powershell
$env:PYTHONPATH='src'
python -m auto_report.cli run-once
```

### 2.2 生成人工复核版

```powershell
$env:PYTHONPATH='src'
python -m auto_report.cli run-once --publication-mode reviewed
python -m auto_report.cli run-once --publication-mode reviewed --reviewer <name> --review-note <text>
```

如需把最新 reviewed 日报同步到飞书文档，并把治理优先队列同步到飞书表格 / 任务：

```powershell
$env:PYTHONPATH='src'
python -m auto_report.cli sync-feishu-workspace --publication-mode reviewed
```

说明：

- `lark-cli` 只用于本地验证、本地美化输出和本地飞书协作同步
- GitHub Actions 继续走仓库内置的 Feishu 官方 API 推送链
- 即使开启 `FEISHU_SIDECAR_ENABLED=true`，在 `GITHUB_ACTIONS=true` 环境里也会自动跳过

### 2.3 补报某一天

```powershell
$env:PYTHONPATH='src'
python -m auto_report.cli backfill --target-date YYYY-MM-DD
```

### 2.4 不重采集，只重渲染

```powershell
$env:PYTHONPATH='src'
python -m auto_report.cli render-report --publication-mode auto
python -m auto_report.cli render-report --publication-mode reviewed --reviewer <name> --review-note <text>
```

### 2.5 重建公开站和内部产物

```powershell
$env:PYTHONPATH='src'
python -m auto_report.cli build-pages
python -m auto_report.cli build-ops-dashboard
python -m auto_report.cli build-source-governance
python -m auto_report.cli build-review-queue
```

说明：

- `out/review-queue/review-issues.json`：报告主题的人审候选
- `out/review-queue/source-lead-issues.json`：discovery/search/governance leads 的人审候选
- `out/review-queue/source-lead-review-status.json`：lead 的轻量状态文件（`approved / rejected / deferred / pending`）
- `out/review-queue/candidate-updates.json`：已批准 lead 转成的 source update 候选集

推荐做法：

1. 先看 `source-lead-issues.json`
2. 再在 `source-lead-review-status.json` 里改状态
3. 重新运行 `python -m auto_report.cli build-review-queue`
4. 最后看 `candidate-updates.json`

当前建议优先处理：

- `anthropic-news`

## 3. 一次运行后先看什么

最重要的结果文件：

- `data/state/run-status.json`
- `data/reports/latest-summary.md`
- `data/reports/latest-summary-reviewed.md`
- `docs/index.html`
- `out/source-governance/source-governance.json`

如果只能看一个文件，优先看：

- `data/state/run-status.json`

因为它能最快回答：

- 这轮有没有真的跑完
- 走的是 `auto` 还是 `reviewed`
- 渠道是成功还是跳过
- 风险等级是什么
- AI 调用和来源健康有没有异常

## 4. `run-status.json` 怎么读

最值得看的字段：

- `publication_mode`
- `review`
- `stage_status`
- `delivery_results`
- `risk_level`
- `ai_metrics`
- `source_health`
- `source_governance`
- `source_stats.report_topics`

快速判断逻辑：

- 这次有没有真正发出去：看 `delivery_results.successful_channels`
- 哪个渠道失败了：看 `delivery_results.failed_channels`
- 是不是 reviewed：看 `publication_mode` 和 `review`
- 来源最近稳不稳：看 `source_health`
- 现在该先治理哪些来源：看 `source_governance`

```powershell
$env:PYTHONPATH='src'
```

先看：

- `data/state/run-status.json > delivery_results`
- `data/state/run-status.json > source_health`
- `data/state/run-status.json > risk_level`

原因通常是：

- `workflow_dispatch` 只跑远端已 push 版本

正确顺序：

1. 本地验证
2. `git push`
3. 再触发线上 workflow

## 7. 本地通过后怎么推仓和触发 GitHub

### 7.1 推仓

```powershell
git add <changed-files>
git commit -m "docs: refresh handoff manuals"
git push origin main
```

### 7.2 手动触发 GitHub Actions

推荐触发 `Collect And Report`，先关闭真实推送：

```powershell
gh workflow run collect-report.yml --ref main -f push_enabled=false -f publication_mode=reviewed -f reviewer=manual-check -f review_note=post-push-verification
```

如果用网页：

- Actions -> `Collect And Report`
- Run workflow
- Branch 选 `main`
- `push_enabled=false`
- `publication_mode=reviewed`

补一条默认规则：

- 触发后先记下 run 链接 / run id
- 不要一直前台 watch 远端跑完
- 默认继续做本地验证、文档整理、下一个任务准备
- 稍后再统一回来核对结果

### 7.3 触发后看什么

至少确认：

- `workflow-guard`
- `test`
- `collect`
- `analyze`
- `report`
- `deploy-pages`
- `ops-dashboard`
- `review-queue`

如果是为了远端验收，不建议第一次就把 `push_enabled=true` 打开。

## 8. 公开站与推送怎么核对

公开站固定入口：

- `https://wsysgz.github.io/VS_AI/`

推送消息固定同时带：

- `公开阅读： https://wsysgz.github.io/VS_AI/`
- `GitHub 原文： <当前 latest-summary 对应链接>`

验证时至少抽查：

- 公开站首页可打开
- Pages 内容与最新报告一致
- `reviewed` 元数据是否正确写入
- 推送中的两个固定链接是否同时存在

## 9. PushPlus / ClawBot 使用提醒

如果你使用的是 `clawbot`：

- 后台必须配置：
  - `PUSHPLUS_TOKEN`
  - `PUSHPLUS_SECRETKEY`
  - 安全 IP 白名单
- 微信侧必须保持 ClawBot 有有效对话上下文

当前仓库已经支持判断：

- `haveContextToken=1` -> ClawBot 当前有有效上下文
- `delivery_status=2` -> 本条消息已成功送达

如果微信收不到，先检查：

1. 最近有没有主动给 ClawBot 发过消息
2. PushPlus 开放接口是否开启
3. `secretKey` 是否正确
4. 安全 IP 白名单是否还是当前公网 IP

## 10. 出问题先查哪里

优先级建议：

1. `data/state/run-status.json`
2. `delivery_results.channels.<name>`
3. `source_health`
4. `docs/index.html`
5. GitHub Actions 最近一次 run
6. `out/source-governance/source-governance.json`

## 11. 值班时最容易踩的坑

- 忘记设置 `$env:PYTHONPATH='src'`
- 只看 workflow 颜色，不看 `run-status.json`
- 本地改了 workflow，却没 push 就去点 `workflow_dispatch`
- 只看 PushPlus `/send` 的返回，不看最终送达状态
- reviewed 轨没关推送，导致把验收结果直接发出去
