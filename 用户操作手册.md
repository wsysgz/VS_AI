### 1.1 LiteLLM Gateway 本机运行

如果你只想一条命令完成“起 gateway -> gateway smoke -> 仓库 smoke -> 自动关 gateway”，直接运行：

```powershell
cd D:\GitHub\auto
pwsh ./scripts/smoke-run-once-via-litellm.ps1
```

默认行为：

- 自动启动本机 LiteLLM
- 先调用 `scripts/smoke-litellm-gateway.ps1`
- 如果检测到 `AI_API_KEY` 或 `LITELLM_MINIMAX_API_KEY`，自动启用阶段 alias 验证：
  - `analysis -> vs-ai-analysis`
  - `summary -> vs-ai-summary`
  - `forecast -> vs-ai-forecast`
  - `prefilter -> vs-ai-prefilter`
- 如果没有 MiniMax key，则回退到 `litellm_proxy + vs-ai-default`
- 默认 `AUTO_PUSH_ENABLED=false`
- 成功后自动关掉 gateway

如果仓库根目录 `.env` 里已经有 `DEEPSEEK_API_KEY`，这一步式脚本可以先直接跑；只有当你要固定自定义 `LITELLM_MASTER_KEY / 端口 / 配置文件` 时，才需要额外创建 `config/litellm/litellm.env.local`。

如果你还没准备本地 LiteLLM 环境，再按下面的分步方式初始化。

推荐本机最短路径：

```powershell
cd D:\GitHub\auto
Copy-Item config/litellm/litellm.env.local.example config/litellm/litellm.env.local
notepad config/litellm/litellm.env.local
pwsh ./scripts/start-litellm-gateway.ps1
```

`config/litellm/litellm.env.local` 至少要填：

```env
LITELLM_HOST=127.0.0.1
LITELLM_PORT=4000
LITELLM_MASTER_KEY=sk-change-me
LITELLM_CONFIG_PATH=config/litellm/litellm-config.local.example.yaml
LITELLM_DEEPSEEK_API_KEY=<your-deepseek-key>
LITELLM_MINIMAX_API_KEY=<optional-minimax-key>
```

说明：

- `scripts/start-litellm-gateway.ps1` 会先读仓库根目录 `.env`，再读 `config/litellm/litellm.env.local`
- 如果 `LITELLM_DEEPSEEK_API_KEY` 为空，但根 `.env` 里已有 `DEEPSEEK_API_KEY`，脚本会自动复用
- 如果 `LITELLM_MINIMAX_API_KEY` 为空，但根 `.env` 里已有 `AI_API_KEY`，脚本会自动复用
- 本地 bootstrap 配置默认只要求 DeepSeek key，就能把 `vs-ai-analysis` 跑起来
- 如果你有 MiniMax key，一步式 smoke 会自动切到 `config/litellm/litellm-config.example.yaml` 做阶段 alias 验证
- 如果你想手动长期起网关并覆盖更多 alias，也可以直接把 `LITELLM_CONFIG_PATH` 指到 `config/litellm/litellm-config.example.yaml`

启动后开新终端做 smoke call：

```powershell
cd D:\GitHub\auto
pwsh ./scripts/smoke-litellm-gateway.ps1
```

如果 smoke 成功，再把主仓库切到 gateway：

### 1.2 GitHub / Repository secrets 怎么配 MiniMax

如果你想让 GitHub Actions 未来也能切到 MiniMax，需要先在 GitHub 仓库里准备统一配置。

进入路径：

- GitHub 仓库
- `Settings`
- `Secrets and variables`
- `Actions`

建议准备：

**Repository secrets**

- `DEEPSEEK_API_KEY`
- `AI_API_KEY`  ← 给 MiniMax-M2.7 或其他第三方 OpenAI-compatible provider
- 保留已有推送 secrets：
  - `PUSHPLUS_TOKEN`
  - `PUSHPLUS_SECRETKEY`
  - `TELEGRAM_BOT_TOKEN`
  - `TELEGRAM_CHAT_ID`
  - `FEISHU_APP_ID`
  - `FEISHU_APP_SECRET`
  - `FEISHU_CHAT_ID`

**Repository variables**

- `AI_PROVIDER`
- `AI_BASE_URL`
- `AI_MODEL`

给 MiniMax-M2.7 的推荐值：

```text
AI_PROVIDER=minimax_svips
AI_BASE_URL=https://api.svips.org/v1
AI_MODEL=MiniMax-M2.7
```

注意：

- 当前 GitHub reusable workflows 已支持统一 provider 配置与 stage-level routing。
- 做远端验收时优先保持 `push_enabled=false`，先核对 `run-status.json` 与 `ai_metrics` 再决定是否开真实推送。

5. 再运行 `python -m auto_report.cli run-watch-checks`
6. 最后看 `candidate-updates.json` 和 `watch-run-results.json`

## 4. `run-status.json` 怎么读

最值得看的字段：

- `publication_mode`
- `review`
- `stage_status`
- `delivery_results`
- `risk_level`
- `ai_metrics`
- `source_health`
- `source_governance`
- `source_stats.report_topics`

快速判断逻辑：

- 这次有没有真正发出去：看 `delivery_results.successful_channels`
- 哪个渠道失败了：看 `delivery_results.failed_channels`
- 是不是 reviewed：看 `publication_mode` 和 `review`
- 来源最近稳不稳：看 `source_health`
- 现在该先治理哪些来源：看 `source_governance`

## 9. PushPlus / ClawBot 使用提醒

如果你使用的是 `clawbot`：

- 后台必须配置：
  - `PUSHPLUS_TOKEN`
  - `PUSHPLUS_SECRETKEY`
  - 安全 IP 白名单
- 微信侧必须保持 ClawBot 有有效对话上下文

当前仓库已经支持判断：

- `haveContextToken=1` -> ClawBot 当前有有效上下文
- `delivery_status=2` -> 本条消息已成功送达

如果微信收不到，先检查：

1. 最近有没有主动给 ClawBot 发过消息
2. PushPlus 开放接口是否开启
3. `secretKey` 是否正确
4. 安全 IP 白名单是否还是当前公网 IP
