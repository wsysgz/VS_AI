# V2升级方案

> 仓库：`D:\GitHub\auto`
>
> 作用：取代 `V1升级方案.md`，承接所有仍有现实价值的长期项，并作为第二阶段升级路线图
>
> 最后更新：2026-04-28

## 1. V2 的起点

V1 到这里已经可以视为工程收口完成：

- 飞书单通道交付稳定
- 公开站 IA V2 已完成
- 来源治理尾项已压缩到持续运营观察
- 发布级本地验证链和远端完整链都已闭环

V2 不再解决“先让系统能跑”的问题，而是解决三个更长期的问题：

1. 让采集与来源治理更稳、更少人工追打
2. 让 AI 运行面更可度量、更可比较、更便于替换
3. 让系统从单仓脚本型工程，逐步演进成可扩展的长期服务

## 2. V2 目标

V2 的目标不是立刻重构，而是在不破坏当前稳定主链的前提下，完成第二阶段升级：

- 更稳的来源入口与阻断证据体系
- 更强的 AI provider 策略、评估与预算管理
- 更成熟的运营台、对比分析和公开站长期能力
- 为未来编排、状态入库、服务拆分建立清晰边界

## 3. 设计原则

- 保留现有主链：`collect -> analyze -> render -> publish`
- 新能力优先 sidecar 化、配置化、可回滚化
- 不为抽象而抽象，先解决稳定性和可维护性
- 所有升级都必须能通过本地验证链独立验收
- 文档、验证、运行态必须跟代码同步推进

## 4. V2 阶段划分

### 阶段 A：来源与采集控制面强化

目标：减少 blocked / fragile 来源反复返工，把证据链、旁路和替代入口标准化。

工作包：

- 把 blocked 来源治理从“单次修复”升级成标准证据链模板：
  - `requests`
  - `curl`
  - browser-backed fetch
  - extractor validation
- 为高价值 fragile 来源建立统一旁路策略：
  - stable feed
  - official API
  - official sitemap
  - repo-local watch
- 把 repo-local watch 消费闭环再收紧：
  - changed
  - blocked
  - ignored-noise
  - candidate update
- 评估并试点 1 个 browser/CLI sidecar：
  - 首选 `OpenCLI` 或等价浏览器辅助方案
  - 只做最难、最值钱的单源 pilot

退出条件：

- blocked 源出现时能稳定产出可复用证据链
- 高价值脆弱来源都有明确首选入口和旁路入口
- watch 结果不再长期堆积为“知道有变化但没人消费”

### 阶段 B：AI 运行面与评估面升级

目标：让 provider、model、预算、fallback 和质量评估从“会切换”升级到“可治理”。

工作包：

- 固化 stage-level routing 策略与文档模板
- 完善 budget / guardrail / backup provider 数据闭环
- 扩大 prompt eval 数据集与回归样例
- 统一 Langfuse 追踪口径：
  - stage
  - provider
  - model
  - token
  - latency
  - fallback
- 为关键输出建立更强的中文约束与质量回归

退出条件：

- 本地和远端都能稳定比较不同 provider 方案
- prompt / model 回归有固定样本，不靠口头判断
- 成本和延迟异常能从运行态直接看出来

### 阶段 C：交付面与运营面升级

目标：把“已有交付”升级成“更好运营、更好消费”。

工作包：

- 飞书运营台只补真正能减人力的小自动化
- 强化 review queue 优先级与 issue 质量
- 继续夯实国内外对比分析：
  - 补齐国内来源空洞
  - 提升 head-to-head 可读性
  - 增强站内 drill-down
- 为公开站补长期跟踪能力：
  - 更稳的专题与趋势入口
  - 更明确的差异提示
  - 更轻量的搜索与归档体验优化
- 引入 DailyOps 式持续优化动作，但只在有清晰收益时做

退出条件：

- 运营动作能通过仓库与飞书协同闭环，不靠口头记忆
- 对比分析不再频繁出现“海外有、国内空”的明显缺口
- 公开站优化以真实消费价值驱动，而不是表面改版

### 阶段 D：架构升级候选

目标：为真正的 V2 架构演进做准备，但不抢前面三个阶段的优先级。

承接自 V1 的长期项：

- 状态入库：`SQLite / Postgres`
- 编排升级：`Dagster` 或等价编排层
- collector / analyzer / publisher 拆分边界
- 更强的多频道、多租户能力
- 更明确的服务化监控与回放机制

准入条件：

- 阶段 A/B/C 已经证明当前单仓主链的稳定性瓶颈不是小修可以解决
- 有明确的数据量、运行频率或协作复杂度在推动架构升级
- 升级能先以 sidecar 或局部模块试点，而不是一口气推倒重来

## 5. V2 的优先顺序

推荐顺序：

1. 阶段 A：来源与采集控制面强化
2. 阶段 B：AI 运行面与评估面升级
3. 阶段 C：交付面与运营面升级
4. 阶段 D：架构升级候选

理由：

- 当前最容易重新拖慢工程的是来源稳定性和长期 watch 消费
- 其次是 AI 运行面的可比较性和预算治理
- 交付和公开站已经可用，应该在稳定基础上做增量升级
- 架构大升级现在有价值，但还不该抢主优先级

## 6. 明确不做的事

- 不回头重新定义 V1 范围
- 不为“看起来更先进”而提前拆服务
- 不在没有真实痛点前，引入数据库、消息队列或复杂编排
- 不恢复已经退役的旧非飞书交付路径
- 不把 GitHub Actions 当主调试回路

## 7. 发布与验收门槛

任何 V2 子阶段进入远端确认前，至少满足：

```powershell
$env:PYTHONPATH = "src"
pwsh ./scripts/check-workflows.ps1 -Profile full
python -m pytest tests -q
python -m auto_report.cli evaluate-prompts --dataset config/prompt_eval/baseline-v1.json
$env:AI_DISABLE_LLM = "true"
$env:AUTO_PUSH_ENABLED = "false"
python -m auto_report.cli run-once --publication-mode reviewed
python -m auto_report.cli build-pages
python -m auto_report.cli build-ops-dashboard
python -m auto_report.cli build-source-governance
python -m auto_report.cli build-review-queue
```

## 8. V2 首批建议

如果下一轮直接开工，建议优先做这三件事：

1. 把 blocked 来源证据链标准化成可复用命令与文档模板
2. 为 watch 消费建立更清晰的“变化 -> 候选更新 -> 应用 / 驳回”闭环
3. 扩一轮 prompt eval / provider 比较样例，把 AI 运行面基线再夯实
